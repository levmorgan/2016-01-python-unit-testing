Intro

  • What is Unit Testing?
    • "A unit test is an automated test that verifies the correctness of an independent unit of code."
    • "A testing unit should focus on one tiny bit of functionality and prove it correct."
    • Two important ideas here:
      1. Automated
        • Write once, run many, many times
        • Easily verify your code before committing to version control or pushing it to a server
          • This can also be done automatically by CI or IDE
        • Run before and after a coding session, so you can be sure that you didn't break anything
        • Make changes to old code without the hassle of figuring out what needs to be manually tested
      2. Units
        • To really be verified by a unit test, the code being tested should be as free from dependencies and side effects as possible
        • This requirement for modular code will make your work more reusable and maintainable
        • Code that is independent enough to be unit tested will be much less likely to cause problems in other areas of your program

Some principles of Unit Testing:

  • Each test should be independent. You should be able to call just one test, or all of them, in any order, and have the results be correct.
  • Try hard to make your tests run quickly - if the tests take too long to run, they may not be run as often as they should be.
  • The first step when you are debugging your code is to write a new test pinpointing the bug. While it is not always possible to do, those bug catching tests are among the most valuable pieces of code in your project.
  • Use long and descriptive names for testing functions. These function names are displayed when a test fails, and should be as descriptive as possible.
  • When something goes wrong or has to be changed, and if your code has a good set of tests, you or other maintainers will rely largely on the testing suite to fix the problem or modify a given behavior. Therefore the testing code will be read as much as or even more than the running code. A unit test whose purpose is unclear is not very helpful in this case.
  • Another use of the testing code is as an introduction to new developers. When someone will have to work on the code base, running and reading the related testing code is often the best they can do. They will or should discover the hot spots, where most difficulties arise, and the corner cases. If they have to add some functionality, the first step should be to add a test and, by this means, ensure the new functionality is not already a working path that has not been plugged into the interface.
